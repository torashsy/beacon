# via-mi

X(Twitter)風のUIで、**複数リンク**と**カレンダーメモ**をひとつの公開ページ
（`/@{handle}`）にまとめる発信ツール。Next.js (App Router) + Supabase。

### 実装メモ（デモからの本番化）

- **認証**: IDを決め、端末のFace ID・指紋・画面ロックなどでパスキーを登録する。
  通常の作成・ログインではパスワードを使わない。Supabase Authがパスキーを検証し、
  アプリ内では失効可能な30日スライド期限のセッショントークンを使用する。
- **復旧**: メールアドレスは作成後に追加できる。未確認の間も利用できるが、
  画面上に「復旧手段が未認証」と表示する。確認後は公開プロフィールに確認済みバッジを表示する。
  このバッジは本人確認ではなく、復旧用連絡先の確認完了を表す。
- **画像**: 編集画面でタップ選択したファイルを `storage.ts` 経由で `avatars`
  バケットへアップロード（`{handle}/av.jpg`・`{handle}/bn.jpg`）。DB には URL のみ。
- **フォロー中**: サーバーに保存し、同じIDでログインした複数端末から共有する。

## はじめかた

セットアップは **[SETUP.md](./SETUP.md)** を参照。実装の進め方は
**[HANDOFF.md](./HANDOFF.md)** のプロンプトを使う。

```bash
npm install
cp .env.local.example .env.local   # Supabase の URL と anon キーを入れる
npm run dev
```

## 公開範囲に関する設計制約

- 未フォローの利用者は、本人から共有された **IDの完全一致** でのみ検索できる。
- 名前・属性・ハッシュタグによる横断検索、利用者一覧、レコメンドは実装しない。
  導入する場合は、公開範囲・通報運用・年齢保護・適用法令を先に再検討する。
- **決済機能を作らない**。「支援」は外部URLを貼るだけ。

## セキュリティの要点

- パスキーの秘密鍵と生体情報は端末外へ送信しない。サーバーは公開鍵を使って署名を検証する。
- IDのみの初回登録は、許可Originとレート制限を持つEdge Functionを経由する。
- 書き込みRPCは失効可能なアプリセッションを毎回検証する。`service_role` キーはブラウザへ渡さない。
- 復旧用メールアドレスはSupabase Authで確認し、確認状態だけを公開バッジに反映する。
  パスキーを失った場合は、ログイン画面から確認済み連絡先を使って新しいパスキーを登録できる。
